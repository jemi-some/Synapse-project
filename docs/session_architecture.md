# 채팅 세션 아키텍처 설계 결정

사용자의 메인 피드를 어떻게 운영할 것인지에 대한 두 가지 설계 방향을 비교하고, 최종 결정을 기록합니다.

---

## 비교 분석

| 항목 | 방향 A: 메인 피드 1개 (슬랙/디스코드형) | 방향 B: 세션 여러 개 (ChatGPT형) |
|:---|:---|:---|
| **비유** | 채널 하나 + 스레드 | ChatGPT 대화 목록 |
| **구조** | 사용자당 하나의 피드가 영원히 유지, 모든 인풋(사진/메모/검색)이 같은 타임라인에 쌓임 | "새 대화" 버튼을 눌러 주제별 세션 분리 |
| **세션 생성** | 최초 로그인 시 1회 자동 생성 | 사용자가 명시적으로 생성 |
| **과거 콘텐츠 접근** | 스크롤 + 시맨틱 검색(RAG) | 좌측 사이드바에서 세션 선택 |
| **깊은 대화** | 특정 메시지에서 **스레드(Thread)** 분기 | 세션 내부에서 대화 지속 |
| **UX 장점** | 직관적. "그냥 던지면 됨" + 스레드로 주제별 심화 가능 | 주제별 정리가 깔끔함 |
| **UX 단점** | 메시지가 무한히 쌓일 수 있음 | 세션을 나누는 행위 자체가 허들 |

---

## 단점 해소: 메신저(카카오톡) 패턴 차용

방향 A의 가장 큰 우려인 **"메시지가 무한히 쌓이는 문제"** 는 카카오톡 등 메신저 앱들이 이미 검증한 패턴을 차용하여 해소합니다.

카카오톡의 1:1 채팅방도 수년치 대화가 하나의 방에 쌓이지만, 아래와 같은 장치 덕분에 불편하지 않습니다.

| 메신저의 해결 방법 | 우리 서비스에서의 적용 |
|:---|:---|
| 날짜 구분선 ("— 3월 9일 토요일 —") | 프론트엔드에서 **날짜 구분선 자동 삽입** |
| 키워드 검색 | **벡터 유사도 검색(RAG)** — 키워드가 아닌 의미 기반 검색 ("작년 바다 사진 찾아줘") |
| 스크롤로 과거 이동 | 동일하게 적용 + **커서 기반 페이지네이션**으로 DB 성능 확보 |
| *(메신저에는 없음)* | **스레드(Thread)** — 특정 메시지에서 분기하여 피드 오염 없이 깊은 대화 |

> 결과적으로 **슬랙의 "피드 + 스레드" 구조**를 뼈대로 가져가되, **카카오톡의 "하나의 끝없는 대화방" UX**로 단점을 보완한 하이브리드 형태입니다.

---

## ✅ 최종 결정: 방향 A (슬랙형 피드 + 메신저형 UX 보완)

기획 의도인 **"사진, 메모, 일기를 툭툭 던지는 하나의 인풋 공간 + 깊은 대화는 스레드로"** 에 가장 부합합니다.

### 전체 플로우 요약

```
사용자 최초 로그인
  └─ 백엔드: "이 사용자의 메인 피드 세션이 있나?" 확인
       ├─ 없으면 → chat_sessions에 1개 자동 생성
       └─ 있으면 → 기존 세션 로드

메인 피드 (세션 1개, 무한 타임라인)
  ├─ 사진 업로드 → 피드에 사진 카드 표시 + 정제한 메타데이터를 통해 장소와 날짜 표시 + 백그라운드 벡터화
  ├─ 텍스트 입력 → MCP(Tool Calling) 의도 판별
  │     ├─ search_memories 호출 → RAG 검색 결과를 피드에 표시
  │     ├─ continue_in_thread 호출 → 스레드 자동 오픈
  │     └─ 도구 호출 없음 → 일반 대화 응답
  └─ 날짜가 바뀌면 → 프론트에서 구분선 자동 렌더링

스레드 (우측 패널)
  └─ 특정 메시지의 parent_message_id를 기반으로 독립된 멀티턴 대화
```

### DB 관점

- `chat_sessions`: 사용자당 **1개** (최초 로그인 시 생성)
- `memories`: 검색 가능한 모든 기억 (사진, 사진+텍스트, 텍스트 메모) — 임베딩은 여기서만
- `chat_messages`: 해당 세션에 모든 메시지가 시간순으로 적재 (화면 표시 전용, 벡터화 없음)
  - `parent_message_id IS NULL` → 메인 피드 메시지
  - `parent_message_id IS NOT NULL` → 스레드 메시지
