## 프로젝트 개요

### 배경

#### 1. 정보의 홍수 속 '자아의 파편화'
오늘날 우리는 매일 수많은 정보를 소비하고 기록하지만, 그 기록들은 노션, 스크린샷, 블로그, SNS 등에 파편화된 상태로 흩어져 있습니다. 이로 인해 내가 진정으로 무엇을 좋아하고 어떤 생각을 거쳐 성장했는지 복기하기가 점점 어려워지고 있습니다.

#### 2. 기록의 사후 활용 부재
많은 이들이 '기록' 그 자체에는 열중하지만, 기록된 데이터 사이의 맥락(Context)을 연결하여 인사이트를 도출하는 단계에는 이르지 못합니다. 과거의 생각과 현재의 고민이 연결되지 못하는 **'단절된 기록'**은 단순한 데이터 적체에 불과합니다.

#### 3. 변화하는 취향의 궤적 추적 필요성
사람의 취향과 가치관은 고정된 것이 아니라 시간에 따라 유기적으로 변화합니다. 하지만 우리는 '언제, 왜, 무엇에' 열광했는지 그 미묘한 변화의 지점을 망각하곤 합니다. 나의 변화 과정을 데이터로 증명하고, 그 궤적 속에서 변하지 않는 본질적인 취향과 강점을 발견할 도구가 필요합니다.

### 핵심 가치

> **흩어진 경험의 조각들을 연결하여 사용자(나)의 잠재적 취향과 사고방식을 발견한다.**

### 대상 데이터

영화/독서/음악 취향, 관심사(관심 분야), 사진 기록, 수집한 문장들, 떠오른 생각, 감정 기복, 브라우저 북마크 등

### 프로젝트 목표

#### 1. 시계열 기반의 취향 아카이빙 (Temporal Preference Tracking)
> "단순한 기록을 넘어, 시간에 따른 사용자의 관심사 변화를 추적하고 시각화합니다. '언제, 무엇에 열광했는가'에 대한 데이터를 통해 개인의 변화 궤적을 객관적으로 증명합니다."

기대 효과: 과거와 현재의 연결을 통한 자아 연속성 확인 및 데이터 기반의 라이프 로그 구축.

#### 2. 데이터 기반의 강점 및 페르소나 도출 (AI-Driven Competency Discovery)
> "사용자가 수집한 문장, 관심사 등 방대한 텍스트 데이터를 벡터 분석하여 반복되는 패턴과 고유한 강점을 추출합니다. 이는 주관적인 자기객관화를 넘어 데이터가 말해주는 '나만의 전문성'을 발견하는 과정입니다."

기대 효과: 커리어 전문성 강화 및 정체성 확립.

#### 3. 비선형적 사고의 연결 (Associative Knowledge Graph)
> "파편화되어 존재하는 아이디어와 메모들을 벡터 공간 상의 유사도를 기반으로 연결합니다. 서로 무관해 보이던 정보들 사이의 맥락(Context)을 AI가 찾아냄으로써 창의적 통찰을 유도하고 '제2의 뇌(Second Brain)' 역할을 수행합니다."

기대 효과: 정보의 유기적 결합을 통한 창의성 극대화 및 지식의 자산화.

#### 4. 메타인지 강화를 위한 인지적 거울 (Cognitive Reflection Mirror)
> "자신의 생각과 행동 패턴을 AI를 통해 객관적으로 관찰함으로써 메타인지(Metacognition) 능력을 향상시킵니다. AI와의 대화를 통해 자신의 편향을 점검하고, 더 나은 의사결정을 내릴 수 있도록 돕는 '인지적 피드백 루프'를 형성합니다."

기대 효과: 자기 성찰의 고도화 및 심리적 성장 지원.

### 핵심 타겟

- 기록은 열심히 하지만 다시 들여다보지 않는 사람.
- 자신의 강점이나 취향을 객관화해서 정리하고 싶은 사람.
- 과거의 생각과 현재의 고민을 연결해 인사이트를 얻고 싶은 사람.

## 페인 포인트 & 문제 정의

### 문제 1: 흩어진 디지털 기록의 맥락 손실

**현상**:
노션, 카카오톡 나에게 보내기, 스크린샷, 물리적 수첩 등 기록의 저장소가 다양합니다.

**문제**:
정작 특정 아이디어나 과거의 생각이 필요할 때 어디에 있는지 찾지 못하거나, 존재 자체를 잊어버려 데이터가 '디지털 쓰레기'가 됩니다.

**결과**: 
과거의 맥락을 되살리는 데 시간과 노력 소모, 결국 포기

### 문제 2: 검색의 한계: 키워드 vs 맥락 (Keyword vs Context)

**현상**:
기존의 메모 앱은 정확한 단어를 기억해야만 찾을 수 있는 '키워드 검색' 방식입니다.

**문제**:
- 정확한 키워드를 기억해야만 검색 가능
- 시간, 장소, 감정을 복합적으로 조합한 검색 불가능
- 모호한 자연어 질의에 대응 불가

**해결**: 
임베딩 모델을 활용한 시맨틱 검색(Semantic Search)을 통해 단어가 달라도 '의미'가 통하면 찾아주는 연결성 제공.

### 문제 3: 자기 객관화의 부재 (Lack of Self-Objectification)

**현상**:
사람은 자신의 변화를 주관적으로 판단하며, 과거의 자신을 미화하거나 왜곡하기 쉽습니다.

**문제**:
내가 실제로 무엇에 시간을 썼고, 어떤 주제에 열광했는지에 대한 객관적 지표가 없어 '메타인지'가 작동하기 어렵습니다.

**해결**:
데이터 시각화와 AI 리포트를 통해 내 관심사의 이동 경로와 반복되는 사고 패턴을 증거(Data-driven)로 제시.

### 문제 4: '구슬'은 많으나 '보배'가 되지 못함 (Insight Gap)

**현상**:
수천 개의 메모가 쌓여 있지만, 그 메모들 사이의 관계를 파악해 새로운 인사이트를 만드는 것은 오롯이 사용자의 뇌에 의존합니다.

**문제**:
파편화된 정보들이 서로 충돌하거나 보완하며 새로운 아이디어가 되는 '창의적 스파크'가 일어나지 않습니다.

**해결**:
LLM이 벡터 DB를 탐색하며 무관해 보이는 정보(예: SF 영화 설정 + 최신 AI 논문)를 연결해 통찰(Insight)을 제안하는 비선형적 사고 지원.

## 만든 이유 & 동기

### 1. 개인적 동기

#### 경험 기반 문제 인식

**실제 경험**:
```
"작년에 친구랑 제주도 갔을 때 그 카페 이름 뭐였지?"
→ 갤러리 수백 장 스크롤
→ 메모 앱에서 "제주" 검색
→ 30분 후에도 못 찾음
→ 포기
```

**깨달음**:
- 디지털 기록은 많지만, **찾을 수 없으면 없는 것과 같다**
- 사람의 기억은 "키워드"가 아니라 "맥락"으로 작동한다
- AI가 이 맥락을 이해하고 찾아줄 수 있다면?

#### 기술적 호기심

**질문**:
```
"최신 AI 기술(RAG, Vector DB, Tool Calling)을 실제 서비스에 어떻게 적용할까?"
```

**동기**:
- OpenAI GPT-4o Vision, text-embedding-3-small 같은 최신 모델 체험
- pgvector, HNSW 인덱스 같은 벡터 검색 기술 실전 적용
- MCP Tool Calling으로 LLM을 "도구를 사용하는 에이전트"로 만들기


## 핵심 기능 및 UX 시나리오 (Core Features)

위에서 정의한 '자아의 파편화'와 '맥락 손실' 문제를 해결하기 위해 다음과 같은 3가지 핵심 기능을 구현했습니다.

### 1. Multimodal RAG Pipeline (기억의 벡터화)
- **문제 해결**: 흩어진 사진과 텍스트 메모를 하나의 의미 공간(Semantic Space)으로 통합합니다.
- **기능**: 사용자가 사진과 메모를 입력하면, OpenAI Vision API가 시각적 요소를 텍스트로 전환하고 이를 임베딩하여 pgvector 모델에 영속적으로 적재합니다.

### 2. Agentic Routing & Semantic Search (지능형 기억 인출)
- **문제 해결**: 단순 키워드가 아닌, '의도'와 '맥락'을 기반으로 과거의 기록을 찾아냅니다.
- **기능**: 사용자의 자연어 입력(예: "작년 겨울 바다에서 느낀 감정")을 LLM이 분석하여 자율적으로 `search` 도구를 호출하고, 코사인 유사도 연산을 통해 최적의 과거 기록을 인출합니다.

### 3. Threaded Conversation (맥락 무손실 꼬리물기 대화)
- **문제 해결**: 단발성 검색에 그치지 않고, 과거의 기억을 바탕으로 깊이 있는 메타인지(자아 성찰) 대화를 유도합니다.
- **기능**: 검색된 과거 기록(부모 메시지)을 Context Window에 명시적으로 주입하여, 메인 피드의 흐름을 방해하지 않고 Slack 형태의 종속된 스레드 영역에서 수십 턴의 깊은 대화를 이어갑니다.

## 시스템 아키텍처 및 기술 스택

이러한 비전을 달성하기 위해, 확장성과 1인 개발 생산성을 최우선으로 고려하여 기술 스택을 구성했습니다.

### Tech Stack
- **Backend**: FastAPI, Python 3.12 (비동기 I/O 기반의 고성능 API 처리)
- **Database**: Supabase PostgreSQL, `pgvector` (시맨틱 검색 및 상태 영속성)
- **AI/LLM**: OpenAI GPT-4o-mini, text-embedding-3-small, 원격 함수 호출(MCP Tool Calling)
- **Frontend**(데모): Vanilla JS (Component Pattern), CSS3, Vite (경량화된 SPA 구현)

### Architecture Decision: Serverless에서 FastAPI로의 전환
초기 MVP는 Supabase Edge Functions(서버리스) 기반이었으나, Multimodal RAG 및 Tool Calling의 복잡한 파이프라인으로 인해 콜드 스타트와 타임아웃(30초 제한) 병목이 지속적으로 발생했습니다. 구조적 한계를 체감하고, 비동기 동시성(Concurrency) 처리에 최적화된 **FastAPI 로 백엔드를 전면 마이그레이션**하여 앱의 응답성과 처리 속도를 크게 개선했습니다.
